FTS5 fragmentation advisory threshold raised 10x → 25x (#1663 v0.87). Dogfood-found within the v0.87 release window itself: shipping #1612's 10x threshold immediately exposed that the config corpus on a 27-project install settles at 10.4x **immediately after** `rebuild_fts` — the advisory false-positived on every doctor inspection, training users to ignore it. Live data point: pre-rebuild config corpus 62.7x → post-rebuild 10.4x (data rows 453k → 15k, idx rows 7235 → 1466). New threshold 25x clears the observed post-rebuild floor with headroom while still catching the 62.7x degradation case that motivated #1612 in the first place. New `TestFTS5FragmentationThreshold_ValueClearsPostRebuildFloor_1663` pins the constant against both bounds so future tightening must ride a new issue + the corpus measurements that justify it.
