**Bench baseline refresh workflow ([#672](https://github.com/kwad77/pincher/issues/672) workstream 2 v0.79 hardening):** new `.github/workflows/bench-baseline.yml` runs `go test -bench` at `-benchtime=10s` against `internal/index` + `internal/server` on a fresh CI runner and uploads the regenerated `testdata/bench/{index,server}.bench.txt` files as an artifact. Triggered manually via the Actions UI `workflow_dispatch` button — operator picks the tag or branch they want to baseline, downloads the artifact, sanity-checks the deltas, copies into the repo, commits. The next `make corpus-bench` run gates against the fresh numbers. Closes the v0.79 prep gap surfaced in the [#672 perf-baseline audit](https://github.com/kwad77/pincher/issues/672#issuecomment-4480258511): the committed baseline was pinned to PR #173 era (pre-v0.71) and drifted 8 feature minors without re-validation, making the "no perf regression >5%" workstream-2 acceptance criterion unenforceable. CLAUDE.md gains a 7th release-prep checklist item — bench baseline decision — that defaults to skip for patch and feature releases, refresh for `.x9` hardening releases or for any release whose perf claims are the rationale. Wrong call (refreshing on every release) silently absorbs regressions; right call (refresh only when justified) keeps the gate enforceable.
