Python AST extraction can route through a persistent subprocess (#1626 v0.87, opt-in). v0.85 dogfood measured Python extraction at ~145ms/file — dominated by the per-file `python3` process spawn + interpreter init (~80ms cold on Windows) rather than `ast.parse` itself. New `pythonRunner` keeps one CPython subprocess alive across many extract calls: `python_extract.py --daemon` runs a newline-delimited JSON request/response loop on stdin/stdout; the Go side serialises requests through a mutex. First request still pays the spawn cost (~140ms) but every subsequent file is a sub-millisecond round-trip. Gated behind `PINCHER_PYTHON_AST_DAEMON=1` for now — the per-file spawn path stays the default until dogfood data confirms the win and the daemon's restart-on-crash / framing-error recovery has soaked. The daemon self-heals: a syntax error in one file returns an error response without killing the subprocess; a broken pipe or framing error resets the runner so the next call relaunches. On a 500-file Django/Flask project the amortised win is the difference between ~72s and ~1s of Python extraction wall-clock.
