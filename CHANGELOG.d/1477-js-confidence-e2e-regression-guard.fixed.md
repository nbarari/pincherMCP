End-to-end regression guard for JavaScript AST extraction confidence (#1477 v0.87). #1477 reported that AST-extracted JavaScript symbols arrived at the regex-tier confidence floor (0.95 / 0.975) in production despite the v0.71 #1354 fix and its passing unit test. The bug was fixed somewhere between v0.73 (filed) and v0.87 — verified via a cross-project pinchQL query showing every normal JS symbol at 1.0 (the only sub-0.99 stragglers are in a `.pnp.cjs` Yarn Plug'n'Play generated file, where regex fallback is correct). But the issue's own root-cause note was right: the existing unit test (`internal/ast/javascript_ast_confidence_test.go`) exercises `extractJavaScriptAST`'s `ConfidenceOverride` in isolation — it never proved the indexer pipeline (langAdapter → Symbol struct → BulkUpsertSymbols → DB) preserves the override end-to-end. This adds `TestIndex_JavaScriptAST_ConfidenceEndToEnd_1477` (a real `.js` file through `Index()`, asserts persisted confidence ≥ 0.99) plus a `.mjs`-extension sibling — the report specifically flagged `.mjs` files as a candidate language-detection misroute. The issue had been orphaned in a never-cut `v0.78.2` patch milestone.
