**`pincher stats` — `retry_rate` → `zero_result_rate` rename + new `retry_success_rate` field ([#1494](https://github.com/kwad77/pincher/issues/1494)):** the JSON field and human-readable label "Retry rate" were both misleading — the value was `queries_zero_result / queries_total`, which on a healthy codebase is dominated by audit-shape queries (`MATCH (f:Function) WHERE NOT EXISTS { (f)-[:CALLED_BY]->() } RETURN f`) where zero rows is the success signal. The honest name is `zero_result_rate`; that's now the JSON field and the displayed label. A new `retry_success_rate` field surfaces `queries_retried_succeeded / queries_zero_result` so users can see how often the agent recovered after a zero-result call — high value here is GOOD (agent self-corrected); low value AT high zero-result-rate is the actual friction signal. Only rendered in the human output when there ARE zero results, to keep healthy projects' stats noise-free. Half 2 of [#1494](https://github.com/kwad77/pincher/issues/1494); Half 1 (split `queries_zero_result` into expected-audit vs unexpected-caller-surprise via parse-time pinchQL classification) needs a schema migration and is queued for v0.80 since v0.78.x is a patch line and v0.79 is reserved for hardening.
