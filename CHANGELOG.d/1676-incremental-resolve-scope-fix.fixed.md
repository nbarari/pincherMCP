Incremental-resolve scope now correctly built from re-extracted files only (#1676 v0.87). The #1629 slices (PRs #1673 / #1674 / #1675) shipped the scoping infrastructure but built the scope set from `seenFiles` — which the walker populates for EVERY file it yields, including hash-skipped ones. On a 1-file edit, `seenFiles` had ~810 entries (the whole project), so the scope-IN-clause matched every pending_edges row and the resolver re-bound the full pool exactly as before. New `extractedFiles` map populated in the main extraction loop right after `totalFiles++` so it only contains files that actually re-extract. Live measurement on the dogfood machine before/after this fix on a 1-file-change tick: resolve_block 810ms → 108ms (7.5×), total tick 1015ms → 320ms (3.2×). pending_in: IMPORTS 2840→30 (95×), CALLS 29564→960 (31×), READS 48317→2511 (19×). Without this fix the entire #1629 umbrella delivered zero practical win — this PR is what actually makes the incremental resolve fast.
