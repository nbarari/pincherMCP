**server (#1085 v0.78):** MCP `notifications/message` emission via `ServerSession.Log`. Pincher now captures the live `ServerSession` at `InitializedHandler` and surfaces background events the request/response envelope can't reach — first user being the schema-drift watcher, which fires a `LoggingLevel="error"` notification before triggering the supervisor respawn so a connected host (Claude Code, Cursor, Zed) can render the reason for the upcoming restart in its UI. The helper (`Server.notifyLog`) is best-effort: it no-ops when no MCP client is connected (dashboard/HTTP-only process shape) and the SDK's client-side level filter gates delivery when the host hasn't subscribed via `logging/setLevel`, so callers never need to know subscription state. New capability tag `mcp_logging` in `_meta.capabilities` advertises the emission lane; the gate test `TestCapability_EveryAdvertisedTagHasRuntimeProbe` pins it against the helper's nil-session no-op behavior, and `notify_log_test.go` covers the wire-level positive + control + cross-check (drift-watcher fires `notifyLog` BEFORE `s.exitFn`, so the message lands on the client before the process exits — pinned via direct ordering observation, not code review). The drift watcher's exit path now goes through `s.exitFn` instead of bare `os.Exit(1)` so tests can observe the notify-then-exit ordering without dying.
