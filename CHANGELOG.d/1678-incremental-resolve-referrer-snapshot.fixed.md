Incremental resolve re-enabled with a correct referrer snapshot (#1678 v0.87). #1679 had gated the #1629 incremental-resolve scoping off after it broke the #457 cross-file CALLS preservation contract — when callee B re-extracts and caller A is hash-skipped, the cascade delete on B's symbols wiped the resolved A→B edge and A's pending_edges were out of scope. This PR fixes it properly: the main extraction loop now snapshots `FilesWithEdgesToFile(relPath)` for each re-extracted file BEFORE spawning that file's goroutine. The query is race-free — a file's incoming edges are only ever cascade-deleted by that file's own goroutine, which hasn't started yet at snapshot time. The resolve scope becomes the union of `extractedFiles` (re-extracted this run) and `referrerFiles` (unchanged files with cross-file edges into a re-extracted file). When the union exceeds 64 files (a hub-file edit whose neighborhood is genuinely large), the scope falls back to the project-wide resolve. Measured on pincher-repo: a leaf-file edit now runs the resolve block in ~87ms (was ~660ms project-wide) — a 7.6× win on the common watcher path, while hub-file edits correctly take the full pass. The #457 contract test (`TestIndex_PendingEdges_PreservesCrossFileCALLSWithoutInvalidation`) passes — the referrer snapshot re-binds the caller's pending_edges.
