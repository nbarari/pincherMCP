`extractGoReads` no longer emits READS pending-edge rows for Go predeclared identifiers (#1622 v0.87). v0.85 observability measured 95% drop rate on READS (48521 of 50989 pending rows on pincher-repo dropped during bind; dominant reason `dropped_to_missing`). The biggest preventable false-positive class: predeclared type and builtin references — `var x int` / `[]string{}` / `x.(error)` / value references to `len` / `append` all walked as bare-ident READS that the resolver would drop because no project Variable named `int` / `error` / `len` exists. `isPredeclaredOrBlank` expanded from the historical 5-name set (`_`, `true`, `false`, `nil`, `iota`) to the full Go spec §6.1 set: 22 predeclared types (`bool` through `uintptr` plus `any` / `comparable`) and 18 builtin functions (`append`, `len`, `make`, `new`, `panic`, etc.). Net effect: fewer pending_edges rows persisted at extract time, less work for `resolveReads`, no behaviour change on real code (the predeclared identifier wins in function scope so the resolver would have dropped these anyway). Trade-off: a project Variable literally named `int` would now be skipped — accepted per the function's doc comment; Go scope rules already prevented that variable from being read locally.
