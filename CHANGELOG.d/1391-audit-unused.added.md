**`audit_unused` MCP tool — Phase 4 composite #3 ([#1391](https://github.com/kwad77/pincher/issues/1391) v0.83):** new dead-code composite with deep-trace confirmation. Runs the existing `dead_code` SQL path then, per candidate, fires a scoped inbound CALLS trace at `confirm_depth` (default 2, max 4) and classifies each candidate by what the trace surfaced: `high` (zero callers — safe to delete with no further confirmation), `medium` (deeper callers — likely a dynamic path the static call graph missed; agent should read the surfaced callers before deleting), `low` (depth-1 caller — SQL-level NOT EXISTS check passed but the recursive CTE found a depth-1 caller, almost always a resolver bug, file an issue rather than delete). Returns `{candidates, summary}` where summary counts the three classification buckets so the agent can spot the bulk of safe-to-delete candidates at a glance. Replaces the N+1 round trips of `dead_code` + per-candidate `trace direction=in depth=2/4` confirmations the agent previously had to fire for each candidate. Read-only; no internal MCP round-trips per the composite contract. Up to 5 caller IDs surfaced in `trace_summary.caller_sample` when 1-5 callers exist so the agent can read them without a follow-up trace. Emits `_meta.warnings_v2.resolver_inconsistency_detected` when the low-confidence bucket is non-empty — the resolver bug should be filed BEFORE acting on the candidate list. 7-test audit suite (positive + negative + control + cross-check) covers high-confidence path, "used helper not in candidates" control, summary-count tally invariant, empty-results empty_reason path, max_results cap, confirm_depth clamp, and gate test. Sequencing: composite #3 of 5 in the Phase 4 roadmap; `onboard_module` and `why_empty` roll forward to v0.84-v0.85.
