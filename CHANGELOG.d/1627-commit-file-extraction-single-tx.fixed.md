Per-file post-extract writes now commit in a single transaction (#1627 v0.86). v0.85 observability showed extraction-phase wall-clock was 73% writer-pool serialization (2064ms of 2840ms on pincher-repo force-reindex was lock contention, not extractor CPU). The previous per-file goroutine took the SQLite writer mutex four times — `ReplacePendingEdgesForFile`, `ReplaceStructFieldsForFile`, `ReplaceInterfaceMethodsForFile`, `SetFileHash` — each a separate `Begin`/`Commit` cycle. New `db.Store.CommitFileExtraction` collapses all four into one transaction. Indexer's per-file path routes through the new method; legacy `Replace*` + `SetFileHash` methods stay for direct callers and tests. Atomicity is strengthened: if any of the four sub-commits fails, every per-file write rolls back together (vs four independent commits previously). On a 786-file project that's ~2358 fewer writer-mutex acquisitions per force-reindex pass.
