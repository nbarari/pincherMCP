**Diagnostic logging + regression tests for Ansible USES_VAR resolve pass ([#1479](https://github.com/kwad77/pincher/issues/1479)):** the reported "USES_VAR shows non-zero in `pincher index --json-summary` but absent from the DB" symptom did not reproduce against the minimal Ansible fixtures the existing `TestIndex_AnsibleUsesVar_BindsAcrossFiles_1165` test exercises, NOR against the new `TestIndex_AnsibleUsesVar_PersistsAcrossForceReindex_1479` (multi-pass force reindex across `group_vars/`, `host_vars/`, `roles/<r>/defaults/` — all three canonical path families produce stable USES_VAR rows across two consecutive force passes) NOR against `TestIndex_AnsibleUsesVar_RoleVarsAndBuildsVarsBind_1479` (the remaining two path families `roles/<r>/vars/` + `vars/`). The resolver works end-to-end on the canonical shapes; the bug must require a specific extractor-emitted edge shape we don't have a fixture for. Pending a reproducer corpus, this slice adds drop-reason diagnostic counters to `resolveUsesVar` — `dropped_from_missing` / `dropped_to_missing` / `deduped_duplicate` — so users hitting the symptom can `slog`-trace whether every pending edge drops because no Setting symbol bound (`dropped_to_missing` non-zero) vs because no from-side anchor was found (`dropped_from_missing` non-zero) vs a clean resolve into BulkUpsertEdges that the DB-level write somehow rejected. Logged at INFO level once per resolve pass and only when at least one edge was dropped or deduped, so healthy projects stay quiet. Plus a docstring on the `DeleteEdgesByKindAndSource` ordering explaining why an empty resolve result is correct (the early-return at the top of `resolveUsesVar` handles the "no pending" case before reaching the delete, so a no-change watcher tick can't wipe the prior edges).
