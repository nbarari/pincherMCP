Windows CI flake fixed in `TestStartAutoIndex_*` (#1605 v0.84). The tests called `srv.startAutoIndex(...)` then asserted state and returned before the fire-and-forget goroutines completed — on Windows the indexer's `<root>/locks/` files were still held when `t.TempDir()` cleanup ran, producing `unlinkat ...locks: The directory is not empty.` New `Server.WaitAutoIndex()` test helper drains the in-flight WaitGroup; both tests now register it via `t.Cleanup`. Production code is unchanged — auto-index goroutines remain fire-and-forget by design (roots/list responds before indexing completes). Observed twice (#1576 + #1602) before the fix.
