C/C++ regex extractor no longer emits phantom Class symbols for struct forward-declarations and struct-typed uses (#1693 v0.89). Fourth fix from the #1389 cross-language false-positive sweep. The C `classRE` (`^\s*(?:class|struct|union)\s+NAME`) matches the column-0 continuation line of a multi-line C prototype — `int connect(int s,\nstruct sockaddr *addr,\nint len)` — where the `struct sockaddr *addr,` line begins with `struct sockaddr`. Every such line emitted a phantom `sockaddr` Class; system headers hit it 5-6× per file and `pincher doctor` reported `qualified_name_collision`. New `dropCTypeUseClasses` filter drops a Class symbol whose declaration line is a forward-decl (`struct Foo;`) or a struct-typed use (`struct sockaddr *addr,` / `struct timeval field;` / `struct Foo arr[10]`) rather than a definition: it scans past the keyword + type name, then on the first decisive punctuation char — `;` `*` `&` `,` `)` `[` `=` → drop; `{` `:` `(` or anything else → keep. Errs toward keep (a real definition wrongly dropped is symbol loss; a type-use wrongly kept is just a recoverable collision). Also resolves the C++ `FSeed`-style forward-decl-then-definition collision since C and C++ share the extractor.
