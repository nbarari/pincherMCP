**HTTP endpoint doc parity test + `/v1/ready` row ([#672](https://github.com/kwad77/pincher/issues/672) workstream 4 v0.79 hardening):** new `TestHTTPRoutes_AllNonToolEndpointsDocumented` asserts every non-tool platform route the server exposes (`/v1/health`, `/v1/ready`, `/v1/dashboard`, `/v1/metrics`, `/v1/openapi.json`, `/v1/events`, `/v1/stats`, `/v1/sessions`, `/v1/projects`, `/v1/index-progress`, `/v1/hook-stats`, `/v1/tool-call-stats`, `/v1/tool-tier-stats`, `/v1/tool-payload-stats`, `/v1/bench-results`, `/v1/capabilities`) has a row in `docs/REFERENCE.md` → ### Additional HTTP endpoints. Caught: `/v1/ready` (#660 — k8s-style readiness probe, sibling of `/v1/health`) was implemented in code and in the OpenAPI spec since v0.58 but missing from the canonical doc table. Users wiring Kubernetes readiness probes against pincher had no doc entry to point at. Inline fix adds the row; the parity test prevents the same drift recurring. `TestHTTPRoutes_ReadyEndpointActuallyServes` pins the runtime-half (200 or 503 on `GET /v1/ready`), matching the existing capability-probe pattern.
