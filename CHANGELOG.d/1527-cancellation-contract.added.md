**Cancellation contract test suite ([#1527 FILE-H](https://github.com/kwad77/pincher/issues/1527) v0.84):** new `internal/server/cancellation_test.go` pins the cancellation behaviour of every "heavy" tool that hits the DB or makes network calls. Each test invokes the handler with an already-cancelled `context.Context` and asserts the handler returns within a tight deadline (≤500 ms for in-process tools, ≤1.5 s for `rebuild_fts` on an empty store). Confirms the existing contract: `handleIndex` propagates ctx down to `idx.Index()` which checks `ctx.Err()` at file-job boundaries (indexer.go:445); `handleFetch` uses `context.WithTimeout(ctx, 15s)` + `http.NewRequestWithContext`, so an already-cancelled ctx aborts the HTTP request immediately; `handleSearch` and `handleTrace` fail-fast on entry validation, so cancellation propagates via the natural error path; `handleRebuildFTS` completes faster than budget on empty stores. Plus a no-transaction-leak check that fires a `handleHealth` call immediately after a cancelled `handleIndex` and asserts the follow-up doesn't block on a leaked writer txn. `TestCancellation_HeavyToolsHaveContractTest` is the gate — adding a new "heavy" tool to `toolComplexityTiers` without a cancellation test fails CI (composites that inherit ctx via their building blocks like search/trace are explicitly exempt, as is the stateless `why_empty`). Future deep interrupt-during-long-op tests live alongside the specific handler (e.g. `internal/index/index_cancellation_test.go`); this PR pins the entry-point + smoke-level contract.
